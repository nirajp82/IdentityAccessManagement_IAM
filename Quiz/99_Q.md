# 📅 10-Day Cloud IAM Architecture Master Plan

## 🚀 Day 1: The Identity Front Door (Authentication & Federation)

**Topic:** B2B Single Sign-On (SSO) and the "Buy vs. Build" philosophy.
**Subtopics:**

* SAML 2.0 vs. OAuth 2.0 vs. OpenID Connect (OIDC).
* The Authorization Code Flow with PKCE (Proof Key for Code Exchange).
* Integrating Identity Brokers (Auth0, WorkOS) vs. Identity Providers (Okta, Entra ID).

**Use Case (Lambda Scenario):**
An enterprise AI startup wants their 50 researchers to log into the Lambda Cloud using their existing corporate Okta credentials so they don't have to manage separate passwords.

**Whiteboard FAQ:**

* **Q: Why use WorkOS/Auth0 instead of building SAML parsing ourselves?**
**A:** Parsing XML-based SAML assertions and maintaining hundreds of IdP integrations is undifferentiated heavy lifting. We *buy* the AuthN gateway (WorkOS) to normalize all incoming enterprise logins into a standard OIDC JSON token, and we *build* the AuthZ layer.
* **Q: What happens if the customer's Okta goes down?**
**A:** Authentication fails, but active sessions (valid JWTs) continue to work until they expire. We cannot authenticate users if their source of truth is offline.

## 🧠 Day 2: Advanced Authorization (The Policy Engine)

**Topic:** Designing the "Build" side of IAM—how to evaluate permissions at scale.
**Subtopics:**

* Role-Based Access Control (RBAC) vs. Attribute-Based Access Control (ABAC).
* Relationship-Based Access Control (ReBAC) & the Google Zanzibar Model.
* Decoupled Policy Engines (Open Policy Agent / OPA).

**Use Case (Lambda Scenario):**
Alice is a "Workspace Viewer" for Project Alpha, but she needs to be temporarily elevated to "Workspace Admin" to spin up 8x H100 GPUs.

**Whiteboard FAQ:**

* **Q: How does our API know if Alice can start a GPU in Workspace B?**
**A:** We use a decoupled AuthZ microservice. The API Gateway sends a permission check (`subject: Alice, action: start_gpu, resource: workspace_b`) to the AuthZ engine, which queries our access graph database and returns an Allow/Deny in <10ms.
* **Q: What is the limitation of basic RBAC here?**
**A:** RBAC lacks context. It says "Admins can start GPUs." It doesn't know *which* workspace the GPU belongs to, or if the customer's billing account is suspended. We need ABAC/ReBAC to factor in resource relationships and dynamic attributes.

---

## 🔐 Day 3: Edge Security, Tokens, & Session Management

**Topic:** Moving identity data securely through a distributed microservices architecture.
**Subtopics:**

* JSON Web Tokens (JWT) vs. Opaque Tokens.
* Asymmetric Cryptography: RS256, Private Keys, and JWKS (JSON Web Key Sets).
* The Token Revocation Problem (Stateless vs. Stateful auth).

**Use Case (Lambda Scenario):**
A compromised user is downloading terabytes of proprietary AI training data. The admin deletes their account, but their session token is still valid for 55 minutes. How do you stop them instantly?

**Whiteboard FAQ:**

* **Q: If JWTs are stateless, how do we revoke them?**
**A:** Pure statelessness is a myth in high-security systems. We use short-lived access tokens (e.g., 15 mins) and a highly available distributed cache (Redis) at the API Gateway. When a user is revoked, their user ID is added to the Redis blocklist. The Gateway checks the blocklist *before* validating the JWT signature.
* **Q: Why not just check the primary database on every request?**
**A:** Database bottleneck. If 10,000 GPUs are making API calls every second, checking a PostgreSQL DB for every token will crash the database. We must push validation to the edge.

## 🔄 Day 4: Directory Sync & Identity Lifecycle (SCIM)

**Topic:** Automating onboarding and offboarding for enterprise customers.
**Subtopics:**

* JIT (Just-In-Time) Provisioning vs. SCIM (System for Cross-domain Identity Management).
* SCIM `/Users` and `/Groups` endpoints.
* Handling synchronization race conditions.

**Use Case (Lambda Scenario):**
A hyperscaler signs a contract and needs to provision 500 AI engineers into Lambda Workspaces. They want this to happen automatically via their Azure AD, before the users even log in for the first time.

**Whiteboard FAQ:**

* **Q: Why use SCIM instead of waiting for the user to log in via SSO (JIT)?**
**A:** JIT only creates the user *after* they log in. If an admin wants to add Alice to a specific GPU workspace today, but Alice hasn't logged in yet, the admin can't find her in the system. SCIM pre-provisions the user so they are fully configured on Day 1.
* **Q: How does SCIM improve security (Deprovisioning)?**
**A:** When an engineer is terminated in the customer's HR system, their Okta instantly sends a SCIM `DELETE` or `PATCH (active: false)` to our API. We immediately kill their active sessions and revoke their cloud infrastructure keys.

## 🤖 Day 5: Machine-to-Machine (M2M) & Workload Identity

**Topic:** How code, scripts, and containers authenticate without human passwords.
**Subtopics:**

* OAuth 2.0 Client Credentials Grant.
* Service Principals vs. API Keys.
* Cloud Workload Identity (AWS IRSA, SPIFFE/SPIRE).

**Use Case (Lambda Scenario):**
A customer runs a distributed PyTorch training job across 100 Kubernetes pods. These pods need to securely pull training data from a private S3 bucket without hardcoding AWS Access Keys in the container image.

**Whiteboard FAQ:**

* **Q: Why are static API keys a bad architecture choice?**
**A:** They don't expire, they get committed to GitHub, and they are hard to rotate.
* **Q: How do we fix this for cloud workloads?**
**A:** We use **Identity Federation**. We link the Kubernetes `ServiceAccount` to an IAM Role using an OIDC provider. The platform automatically injects a short-lived, auto-rotating Web Identity Token into the pod. The code exchanges this token for temporary cloud credentials.

## 🛡️ Day 6: Advanced Authentication & MFA

**Topic:** Protecting the platform from credential theft and phishing.
**Subtopics:**

* Phishing-resistant MFA (WebAuthn, FIDO2, Passkeys).
* Step-up Authentication (Contextual Auth).
* Rate limiting and credential stuffing prevention.

**Use Case (Lambda Scenario):**
An engineer is logged in with a valid session. They click a button to delete a production cluster of $300,000 worth of GPUs.

**Whiteboard FAQ:**

* **Q: How do we secure highly destructive actions?**
**A:** We implement **Step-Up Authentication**. Even if the user has a valid JWT, the API checks the `acr` (Authentication Context Class Reference) claim. If the action requires a hardware key (YubiKey) and the current session only used a password, the API rejects the request and triggers a UI prompt asking the user to tap their YubiKey right now.

## 🗄️ Day 7: Multi-Tenant Database & Schema Design

**Topic:** How to structure your identity data to isolate customers.
**Subtopics:**

* Multi-tenancy models (Siloed vs. Pool vs. Bridge).
* Relational DB schema design for Workspaces, Users, and Roles.
* Row-Level Security (RLS) in PostgreSQL.

**Use Case (Lambda Scenario):**
You need to design the database tables that map Users to Roles to Workspaces, ensuring that Customer A can never accidentally query Customer B's users.

**Whiteboard FAQ:**

* **Q: How do we model a Workspace in the database?**
**A:** A classic relational mapping: `Users` table, `Workspaces` (Tenants) table, and a `Workspace_Role_Bindings` mapping table (User_ID, Workspace_ID, Role_ID).
* **Q: How do we guarantee tenant isolation at the data tier?**
**A:** We implement a "Pool" model using PostgreSQL Row-Level Security (RLS). Every database query is forced to include the `tenant_id` from the user's validated JWT. The database engine physically prevents reading rows that don't match the tenant ID, preventing application-layer bugs from leaking data.

## ⚡ Day 8: High Availability, Scalability, & Disaster Recovery

**Topic:** Distributed systems trade-offs. What happens when things break?
**Subtopics:**

* CAP Theorem in IAM (Consistency vs. Availability).
* Fail-Open vs. Fail-Closed design patterns.
* Caching strategies and cache invalidation.

**Use Case (Lambda Scenario):**
Your custom Authorization microservice database goes down in the US-East region due to a hardware failure.

**Whiteboard FAQ:**

* **Q: Do we Fail-Open or Fail-Closed if the AuthZ engine crashes?**
**A:** In cloud infrastructure, we *must* **Fail-Closed**. If we cannot verify permissions, we deny the request. Failing open would allow anyone to spin up unbilled GPUs or delete other customers' data.
* **Q: How do we mitigate the latency of a database failover?**
**A:** We run the AuthZ database with cross-region Read Replicas. The API Gateway queries a local high-speed cache (Redis) first. If there's a cache miss, it hits the local Read Replica. Write operations (changing permissions) go to the global Primary DB, which asynchronously replicates to the read nodes.

**Topic:** Putting it all together into a single, cohesive architecture drawing.

**Your Whiteboard Checklist to Draw:**

2. **API Gateway** -> Validates JWT, checks **Redis Blocklist**.
3. **WorkOS/Auth0 (External)** -> Handles the SAML/OIDC SSO handshake.
4. **AuthZ Microservice** -> Receives check requests from API Gateway.
5. **AuthZ Database (PostgreSQL/Spanner)** -> Stores the Workspace/Role mapping graph.
6. **SCIM Worker** -> Background service listening to WorkOS to update the database.

## ⚙️ Day 10: Infrastructure as Code (IaC) & Deployment Security

**Topic:** How you actually ship this platform to production securely.
**Subtopics:**

* Terraform / OpenTofu for defining identity infrastructure.
* Crossplane (mentioned in JD) for Kubernetes-native provisioning.
* Secrets Management (HashiCorp Vault).

**Use Case (Lambda Scenario):**
You need to deploy a new identity environment for a dedicated hyperscaler customer without manually clicking around the AWS or Auth0 consoles.

**Whiteboard FAQ:**

* **Q: How do you manage the Auth0/WorkOS configuration?**
**A:** Everything is defined in code using the Auth0 Terraform Provider. Clients, APIs, and rules are checked into Git. We use a tool like Atlantis for GitOps—when a PR is merged, Atlantis automatically runs `terraform apply` to push the changes to the identity provider.
* **Q: Where do the API keys for WorkOS live?**
**A:** Never in code. They are stored in HashiCorp Vault or AWS Secrets Manager and injected dynamically into the CI/CD pipeline at runtime.
